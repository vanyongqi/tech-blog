# AI 在研发工作流里的高强度使用

前阵子我越来越强烈地感觉到一件事。

很多团队其实不是没在用 AI。相反，大家多少都在用。有人拿它查资料，有人让它润色文案，有人拿它补几段脚本，有人让它写点 demo。

但这里面有一个很微妙的分水岭。

有些人用着用着，会觉得这东西真顺手，已经开始有点离不开了。另一些人用着用着，就会得出一个结论，AI 也就那样，时灵时不灵，很多时候还不如自己来。

这两种感受都是真的。

问题在于，它们说的其实不是同一件事。

前一种人，已经开始把 AI 接进自己的工作流了。后一种人，大多数时候还停留在把 AI 当成一个会说话的搜索框，或者一个偶尔能帮上忙的外包实习生。

我自己这段时间把 Claude 和 Codex 用得很重，慢慢有了一个越来越清晰的判断。

真正决定上限的，不是某一句 prompt 写得漂不漂亮。真正决定上限的，是你有没有把模型选择、上下文、仓库规范、执行边界、测试验证这些东西一起搭起来。

说得再直白一点。

AI 在研发里最值钱的地方，不是它能不能一口气写出 300 行代码，而是它能不能在一个真实的软件工程环境里，持续稳定地帮你往前推事情。

这篇文章我想认真聊的，就是这件事。

不是泛泛聊 AI，也不是那种一眼就能看出来是模型堆出来的提示词大全。我更想把 Anthropic 和 OpenAI 官方文档里那些真正重要的信号拎出来，再掺进我自己这段时间的使用手感，聊聊 Claude、Codex，还有一套更像样的研发工作流，应该怎么接起来。

为了避免信息过期，文中涉及模型和产品能力的地方，都是我在 **2026 年 5 月 6 日** 对照官方文档核过一遍之后写下来的。

---

## 先别急着聊 prompt，先聊模型怎么选

我一直不太喜欢那种一上来就问，哪个模型最强。

这个问题听上去很合理，其实工程上没那么有用。因为真实工作里你面对的从来不是一个孤零零的问题，而是一串任务。有的重，有的轻。有的要速度，有的要稳。有的今天就做一次，有的明天会跑一千遍。

Anthropic 在模型概览里其实已经把这个事说得挺清楚了。`Opus` 对应的是更复杂、更重推理的任务，`Sonnet` 更像综合能力最均衡的主力，`Haiku` 则更偏轻量、高速、成本效率。[A1]

所以坊间那句

> Default to Sonnet, optimize with Haiku, escalate to Opus

我觉得大体是成立的。

但我更想补半句，别把它当教条。

这句话真正有价值的地方，不是帮你选一个永远正确的模型，而是提醒你，模型选择本身就是资源调度。你不是在挑一个抽象意义上最聪明的脑子，你是在给一段具体工作流分配合适的算力和成本。

比如你要做的是一批结构化整理、批量归类、格式清洗，甚至是一些相对机械的文档改写，那你一股脑全丢给最重的模型，很多时候就是浪费。

但如果你现在要做的是这些事：

- 读一大坨需求文档，理出真正的问题
- 对两个技术方案做比较
- 判断哪里是风险点
- 在一堆模糊信息里抓出关键约束

那便宜和快，就不一定是第一优先级了。

很多团队一开始用 AI 用不顺，我觉得不是因为 prompt 不够高级，而是因为任务分层这件事压根没做。所有东西都往一个模型里塞，最后不是成本炸了，就是对结果不满意。

## Claude 和 Codex，看起来都能写代码，但根本不是一回事

这也是我最近感受特别深的一点。

很多人会把 Claude 和 Codex 放在一个维度里比，说白了还是在问，谁更强，谁写代码更厉害。

但真到了高强度使用的时候，你会慢慢发现，这个问题本身就有点偏。

Anthropic 文档对 Claude 的强调，一直都很稳定，长上下文、复杂推理、提示工程、Claude Code 里的仓库记忆和工具协作。[A2][A4]  
OpenAI 对 Codex 的定位也很明确，它不是一个单纯给建议的聊天模型，而是一个 coding agent，能进代码库，能运行命令，能根据结果继续修。[O1]

如果把这两者都放在一个真实研发场景里，我现在更愿意这么形容。

Claude 更像一个很能聊、脑子也挺清楚的搭档。你把一团乱麻扔给它，它擅长先帮你把线头找出来。需求到底在说什么，哪些地方模糊，哪两个方案看着都能做但长期成本不一样，哪些 prompt 看似复杂其实根本没必要，Claude 在这些地方通常很顺。

Codex 则更像那个已经坐到你机器前面的人。

它不是跟你讨论应该怎么做，而是真的去看 repo，去读文件，去改，去跑测试，去看哪一步报错，然后再继续修。

这两种能力，当然会重叠一点。但重叠不代表适合混成一个角色。

我现在比较舒服的用法，基本就是把两者硬拆开。

前面那一段更偏思考、澄清、拆解、比较、归纳的工作，我更愿意交给 Claude。后面那一段需要进入工程环境，把事情干到底的活，我更愿意交给 Codex。

这么分不是因为一个聪明、一个不聪明。

而是因为你终于不再逼同一个模型，在同一轮里同时扮演策略师、架构师、实现者、测试员和收尾工了。

这件事一旦想明白，工作流会顺很多。

## 真正拉开差距的，往往不是 prompt，而是上下文

很多文章一写到这里，就会开始讲 prompt 模板。

什么角色设定、任务描述、输出格式、few-shot 例子，当然都重要。我也不是说这些没用。Anthropic 的 prompt engineering 文档本来就一直在强调这些，结构化指令、清晰边界、示例、格式控制，还有用 XML 标签来切分复杂上下文。[A2][A3]

但老实说，真到研发里狠狠干一阵，你会发现 prompt 只是第一层。

后面那层更大的东西，叫上下文。

同样一句，帮我看下这个 bug，到底是让模型瞎猜，还是让模型真的进入问题状态，差别极大。

如果它不知道这个项目是干嘛的，不知道目录结构，不知道测试命令，不知道哪些模块是高风险区，不知道你们团队平时怎么命名、怎么提交流程、怎么跑验收，那它大部分时候只能靠局部线索猜。

而一旦进入猜的状态，再强的模型也会开始飘。

这也是我现在越来越重视 `CLAUDE.md` 的原因。

Anthropic 在 Claude Code 的 memory 文档里把这个机制讲得很直白。`CLAUDE.md` 就是一个项目级、可持续读取的记忆文件，可以把常用命令、代码风格、工作流约定、仓库结构、团队偏好这些东西提前写进去。[A4]

这看上去像个小功能，实际上非常值钱。

因为它把很多原本只能靠口头传递、或者每次开新对话都要重说一遍的东西，变成了模型随手就能读到的工作环境。

我自己的体感是，一旦开始认真维护这种文件，模型的“人味”反而会更强。

不是它更像人了，而是它终于不再像一个毫无上下文的陌生人。它开始知道这个 repo 的脾气，知道你在意什么，知道哪些地方不能乱动，知道代码改完之后下一步该跑什么。

我也会顺手补一些别的东西。

比如 `README.md` 写清楚启动方式，测试文档写清楚命令，部署文档写清楚环境差异，必要的时候再补一个 `AGENTS.md`，把 agent 在这个仓库里能做什么、不能做什么、做到什么程度算完成，尽量讲明白。

`AGENTS.md` 不是某个官方强制要求的标准文件名，我更多把它当成一个团队约定。它的意义不在格式，而在于你终于把很多隐性规则写出来了。

对人有用。

对 agent 也有用。

## 我现在很少让模型上来就写

这是另外一个变化。

以前我也会这样，脑子里有个需求，直接一把扔过去，来，给我写。

有时候也能出结果。但越到复杂任务，越容易翻车。

后来我慢慢发现，很多翻车不是发生在实现阶段，而是发生在更前面一点。

它可能从一开始就误解了需求。  
也可能没意识到系统里某个边界条件。  
也可能没看出这个模块其实是共享路径，改动面比它想象的大得多。

所以我现在更习惯先让模型把脑子亮出来。

先 plan，再 reflect，再 execute。

这套东西听上去有点像方法论鸡汤，但我自己用下来，真的很实。

先让它说，准备怎么做，数据结构怎么定，为什么要这样拆，可能的风险是什么。  
然后再让它自己回头看一眼，这里面有没有明显漏洞，有没有和现有约束打架的地方。  
最后才开始动代码、跑命令、修测试。

Anthropic 和 OpenAI 官方文档虽然写法不一样，但都在反复强调一件事，复杂任务不要妄想一轮拍脑袋做完，阶段化、显式化、可验证，才是更稳的做法。[A2][O2]

这套流程有一个特别朴素的好处。

你终于能在它写代码之前，先 review 它的思路。

很多返工，其实在那一步就能提前砍掉。

## 评估这件事，大家聊得还是太少了

我现在对很多 AI 文章都有点警惕，尤其是那种上来就展示几个神奇案例，然后得出一个宏大结论的。

不是说案例不重要。

而是你如果没有评估，很多看起来很强的东西，根本不知道能不能稳定复现。

Anthropic 在 test and evaluate 那一组文档里专门讲了这件事。[A5]  
OpenAI 的 agent 文档也把 evaluation 放在一个很靠前的位置上，而不是附属章节。[O2]

这个信号其实已经够明显了。

如果连官方都在反复提醒你要评估，那大概率这件事不是锦上添花，而是工程化的硬门槛。

我自己会把评估粗暴地分成两种。

一种是 prompt 和工作流本身的评估。

你别只拿一个顺风案例去试。至少也要看看复杂输入、缺失输入、带干扰的信息、甚至恶意输入下，它会不会开始跑偏。否则你今天觉得这个 prompt 太神了，明天一换场景就废。

另一种，是代码改动的评估。

这里我还是那句话，没有测试，就没有重构。

我知道这句话现在说出来有点像正确的废话，但真用过一段时间 agent 你就会知道，它一点都不废。

AI 最大的好处，是把改动速度提上来了。  
AI 最大的风险，也是把改动速度提上来了。

如果没有测试兜底，它一次性能帮你改十个文件，听上去很爽。可一旦改偏了，它也会一次性把十个文件一起带歪。

所以我现在对完成这件事的定义很窄。

不是它说我改完了，就算完。  
不是它把 patch 打出来了，就算完。  
不是它解释得头头是道，就算完。

跑过测试，结果对，证据在，那才叫完。

如果没有这层东西，AI 在研发里的很多高效率，最后都会变成高波动。

## 如果你真想把 AI 接进研发，不妨先这么试

我不太想在这里给一份那种标准答案式的 checklist，但如果你问我，一个重度研发用户现在该怎么开始把 Claude 和 Codex 用顺，我会建议你从这几步下手。

第一步，不要一上来就追求一个万能 prompt。

先把任务分层。哪些任务更偏理解和判断，哪些任务更偏执行和验证，想清楚之后，你对模型的期待会合理很多。

第二步，把仓库上下文补起来。

`README.md`、测试命令、部署说明、`CLAUDE.md`、必要时补 `AGENTS.md`，这些东西写出来，短期看很土，长期看全是复利。

第三步，让模型先读，再改。

别一上来就是帮我写、帮我重构、帮我优化。先让它把相关文件看完，把思路说出来。这个动作看起来慢，实际上常常是最快的。

第四步，把 Claude 和 Codex 分开用。

拿 Claude 去拆需求、比较方案、打磨 prompt、找文档里的关键信号。拿 Codex 去进 repo、改代码、跑测试、修问题、收尾。

第五步，给它明确的完成标准。

什么叫完成，哪些命令必须跑，输出要长什么样，失败后怎么处理，哪些情况要停下来问人，这些东西你越提前讲清楚，后面越少拉扯。

最后，也是最容易被偷懒跳过的一步，做评估，做验证，保留证据。

别让整个工作流变成一种靠感觉运行的玄学。

## 我最后越来越相信的一件事

我现在已经不太关心 AI 会不会替代程序员这种问题了。

这个问题当然可以聊，但它太大，也太空。

我更关心的是，研发这套活，接下来到底会怎么被重写。

需求怎么澄清。  
方案怎么比较。  
代码怎么改。  
测试怎么触发。  
知识怎么沉淀。  
一个工程师每天最宝贵的注意力，接下来会被放在哪些地方。

如果从这个角度看，AI 更像是在改写研发工作的操作系统。

它不一定把人踢出去。  
但它会把很多原来靠个人记忆、个人体力、个人耐心撑起来的部分，慢慢改造成一套新的协作方式。

在这套方式里，我现在对 Claude 和 Codex 的理解也越来越固定了。

Claude 帮我把问题想清楚。  
Codex 帮我把事情做到底。  
文档、评估和测试，帮我决定这些结果到底能不能进入真实系统。

我觉得真正的差异，从来都不在于你有没有打开过某个模型。

真正的差异在于，你有没有把它接进你的研发工作流。
