# AI 在研发工作流中的高强度使用

## 目标

这不是一场 AI 科普分享，而是一场面向产品、设计、研发极客用户的实战分享：

- 如何选择合适的模型
- 如何通过 Prompt、上下文和约束把模型调成趁手工具
- 如何把 Claude 和 Codex 接入真实研发工作流
- 如何建立测试和验证闭环，避免“看起来写完了，实际上不能用”

---

## 一、模型的选择

> Default to Sonnet, optimize with Haiku, escalate to Opus.
>
> 默认用 Sonnet，用 Haiku 优化成本，遇难事找 Opus。

### 1.1 基本策略

- `Sonnet`：默认主力模型，综合能力、速度、成本比较平衡
- `Haiku`：轻任务、批量任务、成本敏感场景
- `Opus`：复杂推理、难决策、高风险任务

### 1.2 选择模型时看什么

- 任务复杂度
- 响应速度要求
- 成本预算
- 是否需要长链路推理
- 是否需要高质量写作/分析/规划

### 1.3 一个实用判断

不要一上来就追求“最强模型”，而是先问：

- 这个任务值不值得用最贵的模型？
- 是理解问题更难，还是执行动作更难？
- 是一次性深度分析，还是高频批量处理？

## 二、单次交互提示词：输出质量的第一层杠杆

很多人以为 Prompt Engineering 是“会不会说话”，其实更像是“会不会定义任务”。

### 2.1 一个坏例子

```text
帮我写个代码
```

### 2.2 一个更好的例子

```text
用 Go 编写一个可运行函数，功能是计算斐波那契数列的第 n 项。

要求：
1. 使用递归实现
2. 添加缓存优化性能
3. 包含类型注解
4. 编写 docstring 说明函数用法

写完之后，需要你自己调用这个程序进行验证，确保输出结果和预期结果一致。
```

### 2.3 高质量 Prompt 的四个核心组成

- 你是谁：上下文
- 你需要什么：任务
- 怎样才算做好：标准
- 有哪些限制：约束

### 2.4 结论

Prompt 不是提问，而是任务说明书。

## 三、提示词技巧：从“让它回答”升级到“让它交付”

### 3.1 常见高价值技巧

- 措辞清晰，避免模糊动词
- 用结构化指令，而不是一大段口语
- 给示例，降低歧义
- 让模型先思考，再执行
- 明确输出格式
- 要求它自我验证

### 3.2 让模型先出 Plan

可以先定义：

```text
在写代码之前，先列出你的设计思路和实现逻辑。
确认之后，再定义数据结构、主要函数和核心算法。
```

这样做的好处：

- 提前暴露误解
- 更容易 review 它的思路
- 避免直接生成一大段不可控代码

### 3.3 自我修正

```text
反思你的推理是否有漏洞。如果有，请修正。
```

适合用于：

- 方案设计
- 边界条件检查
- 风险判断
- 复杂逻辑实现

### 3.4 严格控制输出格式

```text
只输出 JSON。不要输出 Markdown 代码块标记。不要输出任何解释性文字。
```

这类约束适合：

- 结构化提取
- 接口对接
- 后续自动化处理

## 四、上下文工程：比 Prompt 更重要的能力

随着上下文长度变大，真正的差异不再只是“会不会写一句 Prompt”，而是“会不会组织工作环境”。

### 4.1 上下文里应该放什么

- `Markdown`
- `JSON`
- 代码库
- 需求文档
- 设计说明
- 测试约束
- 业务背景

### 4.2 不要只给任务，要给环境

一个模型如果不知道：

- 项目目录结构
- 编码风格
- 技术栈约束
- 团队规范
- 测试方式

它生成的内容就很容易“像对的，其实不对”。

### 4.3 可以沉淀成固定文件的内容

- `AGENTS.md`
- `CLAUDE.md`
- `README.md`
- 项目开发规范
- 代码风格说明
- 测试执行方式
- 提交流程与验收标准

### 4.4 一个核心观点

高手和新手的差别，往往不在某一句 Prompt，而在喂给模型的上下文质量。

## 五、把 Prompt 当工程来做

Prompt 不是一次性灵感，而是一个可以持续优化的工程流程。

### 5.1 一个常见流程

- 设计
- 开发
- 评估
- 部署

### 5.2 设计阶段

先明确业务目标：

- 要它干什么
- 写代码？
- 分析项目逻辑？
- 做 Code Review？
- 生成设计文档？
- 帮忙拆需求？

### 5.3 开发阶段

可以用 Metaprompting：

让 Claude 帮你写 Prompt，再去服务其他任务。

常见结构包括：

- `Role`
- `Context`
- `Tags`
- `Markdown`
- `Few-shot`

### 5.4 Few-shot 示例要覆盖不同情况

- Ex 1：简单标准情况
- Ex 2：复杂情况，输入很长
- Ex 3：异常情况，输入缺损，输出报错
- Ex 4：干扰情况，输入包含无关信息，输出需要忽略

### 5.5 评估阶段

需要有验证集，而不是靠感觉判断效果。

验证集应该包含：

- 简单输入
- 困难输入
- 恶意输入
- 缺损输入
- 干扰输入

同时要提前定义什么叫“好”：

- 正确率
- 稳定性
- 格式一致性
- 可执行性
- 是否满足约束

## 六、Claude 和 Codex：不是替代关系，而是分工关系

这一部分不是简单比较谁更强，而是看谁更适合放在工作流的哪一段。

### 6.1 Claude 更适合的事

- 长文档理解
- 需求澄清
- 方案设计
- Prompt 设计
- 复杂分析
- 写说明、文档、PRD、总结

Claude 更像一个高带宽的思考伙伴。

### 6.2 Codex 更适合的事

- 进入本地代码库
- 读真实文件
- 修改代码
- 跑命令
- 执行测试
- 根据执行结果继续修复
- 在工程环境里完成闭环

Codex 更像一个可以落地执行的工程代理。

### 6.3 一个推荐协作方式

#### 用 Claude：

- 讨论需求
- 定义方案
- 优化 Prompt
- 产出任务拆解

#### 用 Codex：

- 读 repo
- 改代码
- 跑测试
- 校验结果
- 输出变更说明

### 6.4 一句话总结

- `Claude`：擅长想清楚
- `Codex`：擅长做到底

## 七、Plan -> Reflect -> Execute：避免 AI 直接瞎干

很多失败不是模型不够强，而是用户让它上来就开始写。

### 7.1 推荐链路

#### Plan

- 先列设计思路
- 先列实现路径
- 先定义主要数据结构、函数和算法

#### Reflect

- 检查推理是否有漏洞
- 检查边界条件是否遗漏
- 检查方案是否与约束冲突

#### Execute

- 再开始写代码
- 再开始跑命令
- 再开始出最终结果

### 7.2 为什么有效

- 更容易发现误解
- 更适合多人协作 review
- 可以降低大模型“一步走偏”的风险

## 八、没有测试，就没有重构

> 没有测试就没有重构。

这是 AI coding 时代最重要的底线之一。

### 8.1 为什么这句话更重要了

AI 会显著提升改动速度，但不会自动保证改动正确。

如果没有测试：

- 你不知道它是不是改坏了
- 你不知道它有没有满足边界条件
- 你也不敢放心接收它的改动

### 8.2 推荐做法

- 改代码前先补测试
- 写完后自动执行测试
- 如果失败，要求它先分析失败原因再修复
- 把常用验证流程写成脚本

### 8.3 一个实战原则

不要接受“我已经完成了”的口头结论，要接受“测试通过了”的证据。

## 九、适合分享时强调的几个观点

### 9.1 AI 的上限，不只是模型能力决定

还取决于：

- 你的任务定义能力
- 你的上下文组织能力
- 你的验证能力

### 9.2 真正的提效，不是一次出奇迹

而是：

- 能稳定复用
- 能进入团队流程
- 能沉淀成文档和规范

### 9.3 别把 AI 当搜索框

更好的方式是把它当成：

- 研究助手
- 设计伙伴
- 代码代理
- 自动化协作者

---

## 十、一个可落地的分享收束方式

最后可以把整场分享收束成一句话：

**AI 不会直接替代研发，但会重塑研发的工作方式。**

更具体一点：

- Claude 帮你把问题想清楚
- Codex 帮你把方案做出来
- 测试和验证帮你决定这些结果能不能真的进入生产

真正的差异，不在“你有没有用 AI”，而在“你有没有把 AI 接进你的工作流”。
