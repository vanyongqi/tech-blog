# AI 在研发工作流中的高强度使用：从模型选择、上下文工程到 Claude / Codex 协作

我越来越强烈地感觉到，很多团队并不是“没用 AI”，而是还停留在把 AI 当搜索框、当临时问答工具、当一次性代码生成器。

这会导致一种很常见的错觉：好像模型有时候很强，有时候又很飘。但如果把视角从“单轮回答质量”切到“研发工作流质量”，你会发现真正决定上限的，往往不是那一句 Prompt，而是下面这些东西：

- 你怎么选模型
- 你怎么组织上下文
- 你有没有把规范沉淀进仓库
- 你让模型先想，还是直接开写
- 你有没有评估和验证闭环
- 你是把 Claude 和 Codex 当替代关系，还是分工关系

这篇文章不是 AI 入门，也不是一堆玄学提示词合集。我想做的是一篇偏研究型的工作笔记：先把**官方文档能确认的事实**摆出来，再讲我在研发场景里的**工作流判断**。

为了避免时间漂移，下面涉及模型和产品能力的描述，都以 **2026 年 5 月 6 日** 我核对官方文档时的信息为准。

---

## 一、先说结论：高强度使用 AI，拼的不是“会不会提问”，而是“会不会搭工作台”

如果你是重度研发用户，我现在的经验判断可以先浓缩成 6 句话：

1. 模型选择不是“永远用最强”，而是按任务复杂度、速度和成本做分层。
2. Prompt 当然重要，但真正长期复利的是上下文工程。
3. `CLAUDE.md` 这种仓库级记忆文件非常值钱，因为它能把高频规则从聊天记录里抽出来。
4. `AGENTS.md`、开发规范、测试命令、提交标准这类文件，本质上都是给 agent 准备的工作环境。
5. Claude 更适合做高带宽思考、长文档理解和方案推演；Codex 更适合进入真实代码库执行、测试、修复、收尾。
6. 没有测试，就没有重构；没有评估，就没有稳定的 AI 工作流。

这 6 句话里，前半部分可以直接从官方文档找到支撑，后半部分则是我基于这些能力边界做的工程化判断。

## 二、模型选择：不要迷信“最强模型”，先建立分层策略

### 2.1 Claude 这一侧，Anthropic 官方文档给出的信号是什么

Anthropic 在模型概览里明确把 Claude 模型分成不同能力层级：比如 `Opus` 面向最复杂的推理与高难任务，`Sonnet` 更强调综合能力与日常主力，`Haiku` 更偏速度和成本效率。[A1]

这意味着很多团队里流行的那句：

> Default to Sonnet, optimize with Haiku, escalate to Opus.

并不是完全没有根据。它本质上是在把 Anthropic 的模型分层翻译成工程决策语言：

- 默认主力用 `Sonnet`
- 批量轻任务或成本敏感场景用 `Haiku`
- 遇到复杂规划、难判断、关键任务升级到 `Opus`

但我更想强调一点：**这不是金科玉律，而是一种资源调度策略。**

如果你的任务是：

- 大量文档整理
- 批量结构化提取
- 固定模板生成
- 一次次重复运行的流水任务

那你真正该问的不是“哪个模型最聪明”，而是：

- 这个任务是否值得付更高推理成本？
- 错误代价有多高？
- 是否需要长链路思考？
- 是否会被批量调用上千次？

很多团队在这里最容易犯的错，就是把所有任务都扔给同一个昂贵模型，然后再抱怨成本不可控、速度太慢。

### 2.2 Codex 这一侧，OpenAI 官方文档给出的信号是什么

OpenAI 在 Codex 官方文档里把它定义为面向软件开发的 coding agent：它可以读代码、写代码、运行命令、修复问题，并在真实工程环境里完成任务闭环。[O1]

这跟“普通聊天模型写点代码片段”是两回事。

对研发来说，这里最重要的不是“它是不是也会写函数”，而是：

- 它是否能进入仓库上下文
- 它是否能真实读取文件
- 它是否能执行测试
- 它是否能根据报错继续修复
- 它是否能把任务做到底，而不是只给建议

所以如果把 Claude 和 Codex放在同一个工作流里看，我更倾向于这样理解：

- `Claude` 负责把问题想清楚
- `Codex` 负责把事情做到底

这不是能力高低判断，而是职责分配判断。

## 三、Prompt 当然重要，但真正拉开差距的是任务定义能力

很多人一聊 Prompt Engineering，就容易把它讲成“怎么说话更高级”。我现在越来越不认同这种讲法。

在研发工作里，Prompt 更像一份任务说明书，而不是一个问题。

比如下面这个请求几乎没有操作性：

```text
帮我写个代码
```

它的问题不是“不够礼貌”，而是缺失了执行所需的信息：

- 背景是什么
- 目标是什么
- 约束是什么
- 成功标准是什么
- 怎么验证结果

如果把它改成下面这样，模型的可执行性会高很多：

```text
用 Go 编写一个可运行函数，功能是计算斐波那契数列的第 n 项。

要求：
1. 使用递归实现
2. 添加缓存优化性能
3. 给出清晰的函数注释
4. 补充最小可运行示例

写完后请自行运行验证，确认输出与预期一致，再给出最终答案。
```

这里真正起作用的，不是某个神秘措辞，而是你把任务拆成了 4 个维度：

- 你是谁：上下文
- 你要干什么：任务
- 怎样算做好：标准
- 不能违反什么：约束

Anthropic 的提示工程文档长期强调结构化指令、清晰任务边界、示例和显式格式控制；它还推荐在复杂场景里使用 XML 标签帮助模型识别不同语义区块。[A2][A3]

OpenAI 的官方 agent 文档也把工具、状态、评估和生产化放在同一个体系里，而不是把“会回答问题”当成终点。[O2]

所以我自己的经验结论是：

**Prompt 的本质不是“把话说漂亮”，而是“把任务讲完整”。**

## 四、上下文工程：比 Prompt 更重要，也更容易被低估

### 4.1 长上下文不是拿来灌水的，是拿来降低误解的

Anthropic 文档里对 Claude Code 的描述有一个我很认同的地方：模型的效果高度依赖于它拿到的工作环境，尤其是仓库内持续稳定的上下文，而不只是当前对话。[A4]

这背后的工程含义很大。

一个模型如果只拿到一句“帮我修这个 bug”，它不知道：

- 目录结构
- 技术栈
- 代码风格
- 约定俗成的边界
- 测试命令
- 部署方式
- 哪些文件不能乱动

那它再强，也只能靠猜。

所以我更愿意把上下文工程理解成：

> 不是给模型更多字，而是给模型更好的工作环境。

### 4.2 `CLAUDE.md` 为什么有用

Anthropic 在 Claude Code 的 memory 文档里明确说明了 `CLAUDE.md` 的作用：它是一个可以被 Claude Code 自动读取的持久化项目记忆文件，用来存放常用命令、项目结构、代码风格、工作流约定、用户偏好等信息。[A4]

这是一个非常关键的机制，因为它把“每次对话重复解释”的成本，转成了“在仓库里维护一份持续生效的操作手册”。

我建议一个好用的 `CLAUDE.md` 至少包含这些内容：

- 项目是干什么的
- 主要目录结构
- 本地启动方式
- 测试命令
- 哪些规范必须遵守
- 哪些模块有特殊约束
- 提交前需要做哪些检查

如果你把这些信息都只放在脑子里，模型永远只能从零开始猜你的偏好；如果你把它们写进 `CLAUDE.md`，模型就开始拥有“长期记忆”。

### 4.3 `AGENTS.md` 不是官方标准，但它是很有价值的工程约定

和 `CLAUDE.md` 不同，`AGENTS.md` 并不是某一家官方文档强制要求的格式。更准确地说，它是一种很有用的团队约定：把 agent 该如何在仓库里工作写清楚。

我通常会把这类内容放进去：

- 哪些目录归谁维护
- 哪些改动必须先跑测试
- 哪些命令允许自动执行
- 哪些操作必须人工确认
- 什么叫“任务完成”
- PR / commit / release 的标准是什么

从本质上说，`AGENTS.md`、`README.md`、测试文档、部署说明、代码规范，都是在做同一件事：

**把人类团队的隐性知识，变成 agent 也能消费的显性知识。**

## 五、让模型先想，再做：Plan -> Reflect -> Execute

我现在越来越少让模型“上来就写”。

因为很多严重错误，不是发生在代码生成阶段，而是发生在更前面：

- 任务理解错了
- 边界条件没想到
- 数据结构设计偏了
- 对现有系统关系判断错了

所以我更喜欢把任务切成三步。

### 5.1 Plan

先让模型写出：

- 设计思路
- 实现路径
- 数据结构
- 关键函数
- 风险点

这一步最大的价值不是省代码，而是暴露误解。

### 5.2 Reflect

然后让模型检查：

- 推理是否有漏洞
- 是否遗漏边界条件
- 是否违反已有约束
- 是否存在更小改动面

Anthropic 在提示工程文档里明确提倡让模型分步骤思考、分块处理复杂任务；OpenAI 在 agent 和推理工作流材料里也强调，复杂任务通常需要显式阶段化，而不是一把梭。[A2][O2]

### 5.3 Execute

最后才让它进入：

- 写代码
- 跑命令
- 修测试
- 写变更说明

这套链路听起来变慢了，但在真实研发里往往更快，因为它显著减少了“生成一大坨，再返工一大坨”的情况。

## 六、Claude 和 Codex 的最佳分工：不是替代关系，而是协作关系

### 6.1 Claude 更像高带宽思考伙伴

如果任务更偏这些方向：

- 长文档理解
- 需求澄清
- 方案评审
- 提示词设计
- 多方案比较
- 复杂系统解释

我通常优先用 Claude。

原因不只是“它回答好”，而是它在处理长上下文、连续分析和高层结构化表达时，体验往往更像一个能陪你把问题想透的搭档。[A1][A2]

### 6.2 Codex 更像执行型工程代理

如果任务更偏这些方向：

- 进入真实仓库
- 查文件
- 改代码
- 跑测试
- 根据报错继续修复
- 补丁、提交、验证、收尾

我通常更倾向于 Codex。

OpenAI 官方对 Codex 的定位本身就是 software engineering agent，而不是单纯对话模型。[O1]

这带来的差异是：

- 不是“告诉你怎么做”
- 而是“直接把这件事做完”

### 6.3 我实际会怎么组合它们

一个比较顺手的组合方式是：

#### 用 Claude 做前置认知工作

- 帮我读需求
- 帮我拆任务
- 帮我做方案对比
- 帮我优化 Prompt
- 帮我审查高层设计

#### 用 Codex 做后置执行工作

- 读 repo
- 定位文件
- 修改实现
- 跑测试
- 根据测试结果修复
- 输出改动摘要

这种分工的好处是：你不会逼一个模型同时扮演“高层策略师”和“低层执行者”的全部角色。

## 七、评估：没有验证集，就没有稳定的 Prompt；没有测试，就没有稳定的改动

这是我这两年越来越在意的一点。

很多团队会认真讨论 Prompt，却很少认真做评估。结果就是：

- 某个 Prompt 今天感觉挺好
- 明天换个输入就崩
- 后天换个模型就漂
- 再过一周谁都说不清到底什么叫“效果好”

Anthropic 文档明确建议为提示工程建立 evaluation 过程；OpenAI 的 agent 文档也把“评估 agent 工作流”作为一等主题，而不是附属项。[A5][O2]

### 7.1 对 Prompt 的评估，至少要有一组验证集

我会把样本分成几类：

- 简单标准输入
- 复杂长输入
- 信息缺损输入
- 干扰输入
- 恶意输入

这样做的目的是避免“只在顺风样本里看起来很聪明”。

### 7.2 对代码改动的评估，核心是测试闭环

这里我想把话说得非常重一点：

> 没有测试，就没有重构。

原因很简单。AI 会显著提升改动速度，但不会自动保证改动正确。

如果没有测试：

- 你不知道它是不是改坏了
- 你不知道它有没有覆盖边界条件
- 你不知道它的“完成”是不是幻觉

所以我现在会强制要求：

- 改代码前先确认测试入口
- 写完后必须运行测试
- 测试失败要先解释原因，再修
- 没有验证证据，就不能算完成

这不是保守，而是对高自动化工作流最基本的尊重。

## 八、把方法落进真实研发工作流

如果让我把前面的内容收束成一条可执行链路，我会这样做：

### 第一步：用 Claude 把任务讲清楚

输入给 Claude 的不是一句模糊需求，而是：

- 背景
- 目标
- 约束
- 风险点
- 验收标准

让它先输出：

- 任务拆解
- 方案比较
- 推荐路径
- 风险清单

### 第二步：把高频约束沉淀进仓库

至少把这些文件维护起来：

- `README.md`
- `CLAUDE.md`
- 测试说明
- 部署说明
- 代码规范
- 需要时补一份 `AGENTS.md`

### 第三步：把执行任务交给 Codex

让它：

- 先读文件
- 再做计划
- 再改代码
- 再跑测试
- 再按测试结果修复
- 最后总结变更

### 第四步：为 Prompt 和改动都建立评估

- Prompt 看验证集
- 代码看测试结果
- 工作流看稳定性与复用性

只有这样，AI 才不是一个“偶尔有惊喜的工具”，而是你研发体系里真正可复用的一部分。

## 九、哪些是官方事实，哪些是我的实践判断

为了避免把“文档事实”和“个人经验”混在一起，我把它们分开写一下。

### 9.1 下面这些，官方文档可以直接支持

- Anthropic 明确区分了 Claude 不同模型层级及其定位。[A1]
- Anthropic 提倡结构化提示、示例、显式格式控制和复杂任务分步处理。[A2][A3]
- Claude Code 支持通过 `CLAUDE.md` 提供项目级持久记忆。[A4]
- Anthropic 建议对提示和系统建立 evaluation 流程。[A5]
- OpenAI 官方把 Codex 定义为 coding agent，用于真实软件开发任务。[O1]
- OpenAI 官方强调 agent 工作流、工具调用和评估的重要性。[O2]

### 9.2 下面这些，是我基于这些事实做的工程判断

- 默认用 Sonnet、批量轻任务用 Haiku、难任务上 Opus，是一套好用的资源调度策略，但不是唯一正确答案。
- Claude 更适合“想清楚”，Codex 更适合“做到底”。
- `AGENTS.md` 虽然不是统一官方标准，但在团队协作里非常值钱。
- 没有测试就没有重构，是 AI coding 时代必须坚持的底线。

我认为好的工程文章，不应该把这两类东西混在一起。因为如果没有边界，读者很容易把“个人偏好”误以为“官方最佳实践”。

## 十、最后的判断：AI 不会直接替代研发，但会重写研发的操作系统

我现在越来越不相信“AI 会直接替代程序员”这种简单叙事。

更准确的说法可能是：

**AI 正在重写研发工作的操作系统。**

它改变的不是“还有没有人写代码”，而是：

- 需求怎么被澄清
- 方案怎么被比较
- 代码怎么被生成和修改
- 测试怎么被触发
- 知识怎么被沉淀
- 一个工程师的注意力如何在思考、执行和验证之间重新分配

如果你只是把 AI 当成高级搜索框，那它的价值会很有限。  
如果你开始把模型选择、上下文工程、仓库记忆、agent 分工和测试闭环都搭起来，它就会慢慢从“工具”变成“工作流基础设施”。

我现在对 Claude 和 Codex 的看法，也是这个逻辑下的产物：

- Claude 帮我把问题想清楚
- Codex 帮我把事情做到底
- 文档、评估和测试，帮我判断这些结果能不能进入真实系统

真正的差异，不在于你有没有用 AI，而在于你有没有把 AI 接进你的研发工作流。
