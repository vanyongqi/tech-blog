# 从 ClickHouse 到 Doris：一次分析引擎迁移的完整复盘

![迁移头图](/api/assets/9 "size=full align=center")

把一张仍在持续写入的分析型明细表从 ClickHouse 迁到 Doris，难点从来都不是“把数据搬过去”，而是如何在**历史数据回灌**和**实时数据不停流入**同时发生的情况下，把链路重新接稳。

这篇文章记录的是我一次真实迁移中的判断过程。重点不放在 ClickHouse 和 Doris 的静态对比，而放在更工程化的问题上：我是怎么拆问题的，为什么最终选了这条路径，哪些地方最容易踩坑，以及迁移完成后我又是怎么验收的。

> 说明：文中 SQL、链路图、存储端点、broker、bucket、group id、库表名等信息均做了脱敏处理，只保留迁移思路和工程结构。

> 对我来说，这不是一次“换库”，而是一次对整条分析链路的重新理解。

## 先说结论

- 这次迁移的核心，不是导出导入，而是**边迁边接**
- 最有效的问题拆法，是先把数据分成**冷数据**和**热数据**
- 冷数据走 `ClickHouse -> Parquet -> S3 -> Doris`
- 热数据走 `Kafka -> Routine Load -> Doris`
- 冷热切换不能只靠时间边界，更稳的办法是用 **Kafka offset** 做工程边界
- DDL 迁移不是语法翻译，而是借迁移机会重做一次**表承载设计**

如果要把这次迁移压缩成一句话，我会写成：

> 真正难的不是迁表，而是在实时数据不停流动的时候，把历史回灌、实时续写和查询切换重新接成一条稳定链路。

## 迁移对象到底是什么

这次迁移涉及的是一张用户搜索行为相关的日志表。它不是聚合表，而是一张典型的**分析型明细宽表**：一边挂着用户行为，一边挂着搜索结果明细。

表里既有用户侧标识、访问端信息和租户侧标签，也有查询时间、查询文本、结果维度、证书信息，以及分层标签这类结构化字段。

其中有一个“结果级指纹”字段很关键。它比“请求级查询标识”更适合做跨查询、跨时间窗口的结果归并和热点分析。这一点，后面直接影响了我在 Doris 里的排序键和索引策略。

这张表最常见的查询路径大致有四类：

- 回看单个用户的搜索记录
- 分析用户查询语句与搜索意图
- 看某类搜索结果在时间维度上的变化
- 围绕特定结果维度做过滤、排行和趋势分析

也正因为它同时承载“用户行为”和“搜索结果”两类语义，所以迁移难点从来不只是表大，而是**明细多、维度多、查询路径也多**。

如果只从规模上看，这次迁移面对的也不是一张“小表”：

- 单表容量大约在 **300G 量级**
- 历史数据导出后，一度被切成了 **5 万多个 Parquet 小文件**

这两个数字虽然不直接等于“迁移难度”，但已经足够说明这次工作不是简单的表搬运，而是一次真正的大表迁移和链路重接。

## 第一件事不是迁数据，而是先拆问题

如果一开始就把它定义成“把一张 300G 的表从 ClickHouse 搬到 Doris”，后面基本一定会越做越乱。因为这不是一张静态表，而是一条还在持续写入的活链路。

我一开始先把迁移问题拆成两半：

- **冷数据**：ClickHouse 里已经存在的历史存量
- **热数据**：Kafka 里持续产生、还在不断流入的新数据

这个拆分看起来普通，但后面几乎所有关键决策，都是围绕它展开的：

- 历史数据用什么格式导
- 中间文件落在哪里
- Doris 怎么接实时流
- 冷热边界怎么切
- 如何避免重复和遗漏

说到底，这次迁移真正难的不是“怎么搬一张大表”，而是“怎么边迁边接”。

## 整体方案：冷数据走 S3，热数据走 Kafka

![迁移总流程图](/api/assets/21 "size=full align=center")

最后我采用的是一条并行方案，核心可以直接概括成三部分：

- 冷数据：`ClickHouse -> Parquet -> S3 -> Doris`
- 热数据：`Kafka -> Routine Load -> Doris`
- 切换控制面：围绕 `Kafka offset` 做对齐、追平、对账与切流

这个方案的关键不在“看上去完整”，而在它承认了一个现实：**实时数据不停流的时候，迁移不是一次整表复制，而是一次链路级重接。**

### 为什么冷数据不直接打到 Doris

我最后没有选单次直接导入，而是先把 ClickHouse 历史数据导出成 Parquet，再上传到 S3，最后由 Doris 去拉取。

原因很现实：

- 大表迁移更需要一条**可持久化、可恢复、可重试**的路径
- 中间文件先落稳，后面补导、重试、分批导入都更从容
- 失败恢复时，问题会变成“从哪批文件继续”，而不是“整条链路重新来”

### 为什么热数据单独接 Kafka

热数据我单独拉了一条 `Kafka -> Doris Routine Load` 的实时接入链路。

这样做的收益很直接：

- ClickHouse 迁移期间不需要停写
- Doris 可以从某个阶段开始承接新增数据
- 历史回灌和实时消费可以并行推进

这一步真正解决的不是“怎么接 Kafka”，而是“迁移过程中如何让新数据不掉地上”。

### 为什么我敢单独新建一个 Kafka topic

这一步如果只写成“新建一个 topic 给 Doris 消费”，很容易让人误以为我只是换了个通道，风险自然就会变成“会不会丢消息、会不会把时间戳搞乱”。

我后来比较明确地把这件事想成了三层：

1. **Kafka topic 是消息通道，不是业务时间的来源**  
   这类链路里真正重要的时间，一般都应该来自消息体里的业务时间字段，而不是 Kafka 自己的写入时间。只要 Doris 侧消费时仍然使用消息体里的事件时间或创建时间，新建 topic 并不会天然改写业务时间语义。

2. **新 topic 承担的是“迁移窗口内的新消息”，不是全部历史**  
   历史数据本来就走的是冷数据回灌路径，新的 topic 只是用来承接迁移期间持续产生的增量消息。也就是说，它不是拿来替代历史，而是拿来把冷热边界之后的新流量单独接稳。

3. **真正防止丢数据的，不是“新 topic”这件事本身，而是边界控制**  
   我这次真正依赖的是：历史数据回灌覆盖切换点之前的存量、实时消费追平切换点之后的增量，再配合 offset 对齐和双侧对账，确认冷热边界前后没有断层。topic 只是承载方式，真正决定会不会丢的，还是切换边界有没有控住。

换句话说，新建 topic 这件事本身并不会自动带来数据丢失；真正的风险来自：

- 上游消息有没有稳定写入到新链路
- 业务时间是不是写在消息体里，而不是临时取 Kafka 写入时间
- 冷热边界有没有通过 offset 和对账校清楚

只要这几件事做对了，新 topic 反而更利于迁移，因为它把“迁移期间的新流量”从旧链路里剥离出来了，后面无论是观测、补偿还是切流，都会更清楚。

## 冷热切换为什么不用时间，而用 offset

很多人第一反应会想按时间点切，比如“某天某时之前的数据走历史回灌，之后的数据走实时消费”。这个边界看起来直观，但真正做起来很容易出错。

原因是这类日志链路的时间语义往往不够干净：

- 写入延迟会带来时间重叠
- 补写和重试会打乱自然时间顺序
- 应用时间和消息消费时间可能根本不是同一件事

所以这次我没有简单按时间点切，而是用 **Kafka offset** 做冷热边界。因为 offset 更贴近真实消费位置，是一条更可靠的工程边界。

这件事后面直接降低了两类风险：

- 历史回灌和实时写入重叠导致的数据重复
- 切换瞬间漏掉某段消息导致的数据缺口

## 冷数据主流程：ClickHouse -> Parquet -> S3 -> Doris

历史数据主流程是最先落地的部分，我实际的做法是：

1. 从 ClickHouse 导出历史数据
2. 转成 Parquet 文件
3. 上传到 S3
4. 在 Doris 侧从 S3 读取并写入目标表

在 Doris 侧，我强烈建议先把 S3 当成一张外部表来验证，而不是一上来就导正式表。

先验证“读得出来”，再验证“写得进去”，最后才是“写得对不对”。

例如我会先做类似这样的读取检查：

```sql
SELECT *
FROM S3(
    "uri" = "s3://migration-bucket/history-export/*.parquet",
    "s3.endpoint" = "https://object-storage.example",
    "s3.access_key" = "masked-ak",
    "s3.secret_key" = "masked-sk",
    "s3.region" = "masked-region",
    "s3.root.path" = "history-export",
    "s3.bucket" = "migration-bucket",
    "format" = "parquet",
    "use_path_style" = "true"
);
```

确认 Parquet 文件能被正确读取后，再执行正式导入。

这一段里我最深的体会有两个：

- **字段顺序必须显式确认**。不要假设外部文件里的列顺序一定和目标表一致
- **类型转换必须在导入时处理清楚**。尤其是时间戳转时间字段，后面分区、排序、校验都会依赖它

## 热数据主流程：Kafka -> Routine Load -> Doris

热数据这条线决定了迁移能不能真正闭环。

我最后采用的是 Doris 的 `Routine Load` 去持续消费 Kafka 消息，把迁移期间产生的新数据实时写进 Doris。

示意配置大致是这样：

```sql
CREATE ROUTINE LOAD analytics.kafka_load_to_target
ON query_log
COLUMNS (
    col_event_time,
    col_result_key,
    col_request_key,
    col_record_id,
    col_record_updated_at,
    col_subject_attr_a,
    col_subject_attr_b,
    col_target_name,
    col_target_type,
    col_query_text,
    col_target_host,
    col_actor_id,
    col_label_group_a,
    col_label_group_b,
    col_target_ip,
    col_source_ip,
    col_tenant_marker,
    col_client_fingerprint,
    col_source_region,
    col_network_number,
    col_target_region,
    col_flag_a,
    col_flag_b,
    col_flag_c,
    col_name_fallback,
    col_total_count,
    col_cert_name_fallback
)
PROPERTIES (
    "format" = "json",
    "desired_concurrent_number" = "1"
)
FROM KAFKA (
    "kafka_broker_list" = "broker-a:9092,broker-b:9092",
    "kafka_topic" = "masked-topic",
    "property.group.id" = "masked-consumer-group"
);
```

但这里很容易有一个错觉：`Routine Load` 开起来了，就等于实时链路接好了。真实情况不是。

真正容易出问题的，反而是这几类细节：

- JSON 字段映射是否和目标表严格对齐
- 数组字段、层级字段是否能被正确解析
- 消息结构变更后，Routine Load 是否会静默出错
- 任务状态有没有持续推进

迁移期间，我至少会持续盯下面这些信息：

- 任务是否暂停
- 状态变化原因是什么
- 消费位点有没有持续推进
- 有没有明显错误或阻塞

最基础的检查方式就是：

```sql
SHOW ROUTINE LOAD;
```

## DDL 迁移不是翻译，而是重设计

![DDL 重设计对比图](/api/assets/11 "size=full align=center")

迁移前，我也一度想过把 ClickHouse 的 DDL 直接“翻译”成 Doris 版本。现在回头看，这种思路最多只能帮你完成第一步，远远不够。

因为一张表在不同引擎里的最佳承载方式，本来就不一定一样。

### 我最初关注的是兼容

最开始我优先做的是字段级兼容：

- `String` 对到 Doris 字符串类型
- `DateTime` 对到 `DATETIME`
- `UInt64`、`UInt32` 对到对应整型
- `Array(String)`、`Array(UInt64)` 对到 Doris 原生数组类型

这个阶段的目标不是最优，而是先保证数据能落进来。

### 真正跑起来后，我开始按查询路径重调

后面我逐渐意识到，沿用 ClickHouse 里的承载方式并不合理。

对这张表来说：

- “请求级查询标识”更像一次查询动作本身
- “结果级指纹”对应的是结果实体本身
- 我的实际分析场景，更多围绕 **“结果级指纹 + 时间”** 展开

所以后面在 Doris 里，我把关注点从“沿用请求级标识”逐步转向了“围绕时间列 + 结果级指纹去优化查询路径”。

这次调整里，我最看重的是四件事：

| 设计点 | 初始思路 | 调整后思路 | 调整原因 |
| --- | --- | --- | --- |
| 排序键 | 围绕请求级标识 | 围绕时间列 + 结果级指纹 | 更贴近真实查询路径 |
| 分桶方式 | 围绕固定哈希键 | `RANDOM` | 降低数据倾斜和固定键约束 |
| 索引策略 | 默认继承原表思路 | 强化结果级指纹过滤能力 | 业务里更常见的是实体级等值过滤 |
| 存储策略 | 先能用 | 压缩、Compaction 按日志型明细表调优 | 兼顾写入稳定和中后期存储效率 |

这也是我对这次迁移最重要的认知变化之一：

> DDL 迁移很多时候不是语法转换，而是借迁移机会重新定义表的承载方式。

### 基于这张 Doris 表定义，我觉得还应该明确提醒的几个注意事项

除了上面的方向调整，这张 Doris 表定义里还有几处很值得单独说清楚。它们不一定是“亮点”，但很容易决定后面链路是否稳定。

#### 1. `DUPLICATE KEY` 不是幂等保障

这张表最终用的是 `DUPLICATE KEY(created_at, fid)`。这更像是为了贴合明细日志场景和查询路径，而不是为了做严格去重。

这意味着一件事：**不要把 Doris 这层的 key 设计误解成幂等语义本身**。  
真正防止历史回灌和实时接入重叠出错的，还是冷热边界控制、offset 对齐和双侧对账，而不是指望 `DUPLICATE KEY` 自动帮你兜住重复。

#### 2. 表分区粒度和导出文件粒度不要混在一起看

这张表在 Doris 里是按月自动分区：

- `AUTO PARTITION BY RANGE (date_trunc(created_at, 'month'))()`

但我在迁移过程中把 Parquet 文件从“按月切分”改成了“按天切分”。

这两件事并不冲突：

- **表分区粒度** 是为了控制 Doris 里的存储与查询组织
- **文件切分粒度** 是为了控制迁移过程里的排障、补导和对账效率

回头看，按天切文件更适合迁移，因为一旦某天数量不对，更容易快速缩小排查范围；而表仍按月分区，则更符合这类日志型分析表的长期查询组织方式。

#### 3. `DISTRIBUTED BY RANDOM BUCKETS 3` 的重点是先求稳，再求细调

这张表最后没有继续用固定哈希键，而是用了随机分桶。

我理解这里最重要的取舍是：**先减少数据倾斜风险，避免过早把分桶策略绑死在某个字段上**。  
对于仍在调整查询路径和迁移链路的场景，这种做法通常更稳，尤其适合先把链路跑通，再看后面的热点分布和查询模式是否需要继续细调。

#### 4. `bloom_filter_columns = "fid"` 是个很业务化的选择

这里不是随手加一个 Bloom Filter，而是明确围绕“结果级指纹”这个字段来的。

如果你的核心过滤路径经常落在某个实体级字段上，那 Bloom Filter 才有意义；如果查询主要走别的组合维度，这个配置就未必是收益最大的。

换句话说，**Bloom Filter 配在哪个字段上，本质上是业务查询路径的选择题，不是 Doris 参数模板题。**

#### 5. 数组字段迁移时，别只盯“能不能建出来”，还要盯“能不能稳定写进去”

这张表里有多组数组字段。表能建出来不代表链路就稳了，真正的压力点在于：

- 上游消息里的数组结构是否稳定
- JSON 到列映射时是否会出现层级错位
- 下游查询是否真的会高频使用这些数组字段

也就是说，**数组类型的风险往往不在 DDL，而在导入链路和消费链路。**

#### 6. `time_series` compaction 和压缩参数，说明这张表本质上就是日志型明细表

这张表最终配了：

- `compression = "ZSTD"`
- `compaction_policy = "time_series"`
- `enable_single_replica_compaction = "true"`

这些配置背后的信号其实很明确：这张表不是静态维表，也不是强更新型表，而是以持续写入、按时间累积、以分析查询为主的日志型明细表。

所以这里真正重要的不是“参数名要背下来”，而是：**你要先承认这是什么类型的表，再去选参数。**

#### 7. `min_load_replica_num = 1` 这类参数，本质上是在迁移阶段给导入成功率留弹性

从迁移视角看，这种参数往往不是为了追求最理想的副本一致性表达，而是为了在迁移窗口里先保证导入链路不要因为局部副本状态而频繁失败。

这类配置值值不值得保留到长期生产，要看后面系统稳定后的要求；但在迁移阶段，它的意义更多是：**先把批量导入链路跑稳。**

## 这次迁移里最容易卡住进度的 5 个坑

### 1. 小文件不是小问题

为了方便导出和恢复，我把历史数据切成了很多 Parquet 文件。思路本身没错，但文件切得太碎，最后变成了五万多个小文件。

真正拖慢导入的，不是总数据量，而是大量小批次导入带来的调度和执行开销。

这个坑让我非常清楚地记住一件事：**前一步里一个看起来只是“参数没调好”的细节，到了后一步，很可能就会变成整个链路的瓶颈。**

### 2. 字段顺序错了，未必会报错

`INSERT INTO ... SELECT ... FROM S3(...)` 这种方式很直观，但只要列顺序错了、时间转换错了，问题不一定表现成显式失败。

更麻烦的情况是：数据能导进去，但导进去之后就是错的。

这种问题往往比“直接失败”更难排查。

### 3. Routine Load 很好用，但不会自动替你对齐结构

只要消息结构里有 JSON、数组、层级字段，你就不能期待它“自动对齐”。

如果这一步没在迁移前期做足验证，很多问题会在数据已经流起来之后才暴露。

### 4. 对唯一性的误判会把后面所有判断带偏

“请求级查询标识”这个字段很容易被误解成单行唯一标识，但实际不是。它代表的是一次查询动作，而一次查询通常会返回多条结果，所以同一个请求级标识会对应多行数据。

这个认知如果一开始没立住，后面无论是幂等处理、冷热切换，还是重叠区去重，都会很容易想错。

### 5. 很多问题不是设计阶段暴露的，而是跑起来之后暴露的

这次过程里，我最认同的一句话就是：

> 完成胜于完美。

不是说设计不重要，而是很多工程问题只有在主流程真的跑起来之后，才会从“可能有风险”变成“这里就是会出问题”。

## 我是怎么确认迁移成功的

对我来说，迁移成功不能只看“Doris 里已经有数据了”。真正让我愿意确认这次迁移已经收口的，是下面几类检查都过了。

```text
迁移验收 = 数据量对齐
       + 样本记录对齐
       + Kafka 消费追平
       + 线上查询平稳切换
```

### 1. 先看数据量是否对齐

我会按时间范围去比 ClickHouse 和 Doris 两边的总量，先确认大的量级没有偏差。

### 2. 再抽样查具体用户记录

因为这张表承载的是用户搜索日志和搜索结果明细，所以我会直接抽用户样本，对比关键字段和结果集是否一致。

这一步的目的，是确认“不是只有总量对了，而是具体场景也对了”。

### 3. 看 Kafka 消费是否真正追平

热数据是通过 `Routine Load` 接入的，所以只要 Kafka 消费没有追平，实时链路就不能算真正稳定。

### 4. 最后看线上查询是否平稳切走

这是最接近业务验收的一步。前面对账做得再漂亮，如果线上查询切到 Doris 之后异常频发，那这次迁移依然不能算完成。

## 最后复盘：这次迁移真正迁走的是什么

迁移完成之后，最直接的收益当然是技术栈统一了，数据平台维护简单了，后续开发和分析路径也更集中了一些。

但如果只从“表迁过去了”来理解这件事，我觉得会低估这次过程的价值。

真正让我印象最深的，不是某条 SQL 终于跑通，也不是某次导入终于成功，而是我对整条数据链路的理解比开始做之前清楚得多：

- 哪些字段才真正影响查询路径
- 哪些边界最容易出问题
- 哪些步骤看起来只是工程细节，实际上会决定整体效率

如果让我把这次复盘压缩成 3 句话，大概会是：

1. **迁移不是搬数据，而是重新梳理链路。**
2. **很多问题不会在设计阶段彻底暴露，只有跑起来才会出现。**
3. **同一张表在不同引擎里的承载方式，未必应该完全一样。**
